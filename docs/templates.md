# Templates

The whole template surface ‚Äî the `render_template` MCP tool, `list_templates`, the skill's catalog + routing map, the Brand Site showcase ‚Äî derives from **two files**:

1. `templates/<key>.html` ‚Äî the template itself
2. `templates.meta.ts` ‚Äî the registry entry (output, dims, vars, **prompts**, tags)

Edit those two files, run `npm run build:skill`, and every downstream surface picks it up automatically.

## The propagation chain

```
templates.meta.ts (+ templates/<key>.html)
       ‚îÇ
       ‚îú‚îÄ‚Üí MCP tools (live import ‚Äî listTemplates.ts, renderTemplate.ts)
       ‚îÇ
       ‚îú‚îÄ‚Üí Skill catalog + routing  (auto-regenerated by scripts/build-skill-catalog.ts
       ‚îÇ                              into skill/escape-velocity-brand/SKILL.md
       ‚îÇ                              every time build:skill runs)
       ‚îÇ
       ‚îî‚îÄ‚Üí Brand Site /brand/social  (auto-discovered from templates/social/*.html
                                       via site/_data/social.cjs filesystem scan)
```

**Never hand-edit SKILL.md to reference templates** ‚Äî the catalog + routing blocks are between `<!-- AUTO-GENERATED:CATALOG -->` / `<!-- AUTO-GENERATED:ROUTING -->` markers and get rewritten on every build. Edit the registry instead.

## Adding a new template ‚Äî step by step

### 1. Create `templates/<key>.html`

Source tokens via `{{ TOKENS_CSS | safe }}` (mandatory ‚Äî see [rules.md ¬ß Token-Sourcing Rule](rules.md#token-sourcing-rule)). Use Nunjucks vars (UPPERCASE convention: `HEADLINE`, `BODY`, etc.). Pattern-match an existing template ‚Äî `templates/social/announcement.html` is a good reference for content-flexible PNG templates.

### 2. Register in `templates.meta.ts`

Add an entry keyed by the path (without `.html`):

```typescript
'social/linkedin-post-portrait': {
  output: 'png',
  description: 'LinkedIn portrait feed post (1080√ó1350). Content-flexible: eyebrow + headline + body + CTA.',
  dims: { width: 1080, height: 1350 },
  optional: ['EYEBROW', 'HEADLINE', 'BODY', 'CTA'],
  tags: ['social', 'linkedin', 'feed-post'],
  prompts: ['LinkedIn post', 'LinkedIn shareable', 'LinkedIn portrait', /* ... */],
},
```

**`prompts` is the single most common omission.** Without it the template is invisible to the skill's routing map ‚Äî the LLM won't know to choose it, and the regression we just diagnosed (custom HTML when a template existed) recurs.

### 3. Run `npm run build:skill`

This regenerates the catalog + routing blocks in SKILL.md from the registry (via `scripts/build-skill-catalog.ts`), then packages the `.skill` zip. Commit the resulting SKILL.md diff alongside your registry change ‚Äî the diff is the proof that propagation worked.

### 4. Add Brand Site coverage

Per the [Brand-Site Coverage Rule](rules.md#brand-site-coverage-rule), every new template MUST appear on the Brand Site.

For social templates: add a `SOCIAL_META` entry (title / size / cmd copy) to `site/_data/social.cjs` keyed by the template slug. The `.njk` page auto-discovers via filesystem scan, and `SOCIAL_META` provides the human-readable metadata. Then run `npm run build:assets` to regenerate the showcase raster into `assets/raster/<slug>.png` (or `<slug>-sample.png`).

### 5. Smoke test

Render a sample via the CLI shim:

```bash
npx tsx generators/image.ts --input templates/social/<key>.html --type html \
  --width <w> --height <h> --var "HEADLINE=Test" -o /tmp/test.png
```

Inspect the PNG. Or add a fixture to `tests/mcp/fixtures/` and run `npm run test:mcp`. See [testing.md](testing.md).

## Modifying an existing template

- **Visual change** ‚Üí edit `templates/<key>.html`. No registry change needed.
- **New var, dimension change, new prompt phrasing the LLM should match on** ‚Üí edit `templates.meta.ts`, then `npm run build:skill`.

## Template variables ‚Äî Nunjucks conventions

- **Uppercase** for caller-supplied vars: `HEADLINE`, `EYEBROW`, `BODY`, `CTA`, `TITLE`, `SUBTITLE`, etc.
- **Lowercase** for auto-injected vars: `tokens_css` (no ‚Äî actually `{{ TOKENS_CSS | safe }}`), `FONTS_URI`, `BRAND_DIR`. The render core injects these regardless of template.
- Document-type templates receive structured args via `_recipient.html` partial and the `core/document.ts` pipeline; see [`_base.html`](../templates/_base.html) and [`_recipient.html`](../templates/_recipient.html).

## Document vs social/carousel distinction

Document templates (`letter`, `offer`, `invoice`, `tos`, `report`) are special:

- They go through the full `core/document.ts` pipeline (markdown body ‚Üí `renderDocumentHtml` ‚Üí `htmlToPdf` with branded footer + watermark + recipient block).
- They take **structured args**: `markdown` (body), `recipient`, `date`, `ref`, `subject`, `confidential`, `lang`, `attachPath`.
- The `render_template` tool dispatches to the document pipeline when the template key is in the document set; otherwise falls through to generic PNG/PDF rendering.

Social and carousel templates are simpler:

- Generic Nunjucks render ‚Üí PNG (via Playwright screenshot) or PDF (via `renderHtmlToPdf`).
- Take `vars` (a flat string-to-string map) + optional `markdown` body if the template has `acceptsBody: true`.

## Anti-patterns

- ‚ùå **Hand-editing SKILL.md** to add a template name or routing row. The next `build:skill` overwrites the marker blocks. Edit the registry; the skill picks it up automatically.
- ‚ùå **Adding a template without `prompts`.** It will be callable via `render_template` (which still works by template key) but **invisible to the skill's routing table** ‚Äî the LLM will fall back to custom HTML. This is exactly how the LinkedIn-post regression slipped in.
- ‚ùå **Adding a template without Brand Site coverage.** Future humans (including you in 3 months) won't find it. See [rules.md ¬ß Brand-Site Coverage Rule](rules.md#brand-site-coverage-rule).
- ‚ùå **Adding the same conceptual asset under multiple keys** (e.g. `social/linkedin-post` + `social/li-post-portrait`). Pick one canonical key per (output, dims, intent) tuple.
- ‚ùå **Hardcoding tokens in template CSS.** See [rules.md ¬ß Token-Sourcing Rule](rules.md#token-sourcing-rule).
